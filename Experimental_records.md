## 2. 项目初始目标与后续调整

### 2.1 初始项目背景

`catch_it` 是一个基于 MuJoCo 和 PPO 的机器人强化学习项目。  
原始项目同时包含：

- `Tracking` 任务
- `Catching` 任务
- 原有移动底盘 + 机械臂 + 手部的机器人模型

项目主线是：

1. 环境随机抛出物体
2. 机器人根据观测输出动作
3. 环境推进物理仿真
4. 根据动作结果计算奖励
5. PPO 反复采样和更新策略

### 2.2 当前聚焦的目标

在后续工作中，项目目标逐步收敛为：

- 只做 `Tracking`
- 不再做 `Catching`
- 将原来的机器人模型替换为新的 `tidybot`
- 目标由“抓取物体”简化为“让夹爪末端接近并碰到飞来的物体”

这样调整的原因主要有三点：

1. 原项目逻辑较复杂，作为入门更适合先集中在 `Tracking`
2. 新模型 `tidybot` 更贴近后续真实平台
3. 先把“追踪接近”学稳定，再考虑更难的抓取闭环

## 4. 环境与训练基础问题排查

### 4.1 基础依赖问题

最开始训练时，曾出现：

```text
ModuleNotFoundError: No module named 'hydra'
```

这说明运行环境缺少 `hydra`。  
后续确认项目应在 `conda` 的 `dcmm` 环境中运行。

### 4.2 并行环境报错

后续又出现过：

```text
AttributeError: 'DcmmVecEnv' object has no attribute 'random_delay'
```

这类错误说明：

- 环境类在并行 worker 中初始化不完整
- 某些运行时属性没有在构造或 reset 阶段正确设置

该问题后来被修复，训练可以正常进入采样阶段。

### 4.3 TensorBoard 查看问题

后期查看曲线时还遇到过 TensorBoard 插件兼容报错，典型现象是：

- TensorBoard 页面打不开或部分曲线异常
- `hparams` 插件报 `MessageToJson` 相关错误

后续主要转而使用：

- `wandb` 离线日志
- TensorBoard 的 scalar 曲线
- 命令行输出

来综合判断训练效果。

## 5. 模型替换：从旧机器人迁移到 tidybot

### 5.1 新模型引入

后续将项目中的机器人主体替换为：

- `assets/urdf/stanford_tidybot2/tidybot.xml`

新的机器人由以下部分构成：

- 移动底盘
- 7 关节机械臂
- 夹爪末端

### 5.2 迁移思路

此次迁移不是简单替换 XML，而是把整个项目逐步调整为适配 tidybot：

1. 只保留 `Tracking` 入口
2. 删除或弱化旧机器人 / Catching 的运行分支
3. 重建 tidybot 的观测空间
4. 重建 tidybot 的动作空间
5. 调整控制方式
6. 重新设计奖励函数和成功判定

### 5.3 迁移后当前任务定义

目前的 tidybot Tracking 任务可以概括为：

- 机器人面对来球
- 底盘可以运动
- 机械臂 7 个关节可运动
- 夹爪末端尽量追踪并接触飞来的物体

与原项目相比，当前任务更强调：

- 移动底盘与机械臂协同
- 末端追踪精度
- 训练稳定性

## 6. 控制结构与动作设计调整

### 6.1 动作维度调整

当前 tidybot Tracking 任务的动作维度为：

- `act_dim = 9`

动作组成如下：

- 底盘 `base(2)`
- 机械臂 `arm(7)`

即：

- 2 维用于底盘平面运动
- 7 维用于 7 个关节的增量控制

### 6.2 控制方式变化

旧项目中部分控制较依赖：

- IK
- PID

在 tidybot Tracking 任务中，后来逐步转为更直接的关节空间控制：

- 底盘：平面速度控制
- 机械臂：7 关节目标增量控制
- 手部：保留 1 维接口，但在 Tracking 中不是主要学习对象

### 6.3 相关问题

在测试中曾观察到：

- 底盘几乎不动
- 机械臂主要是上部几个关节在动
- 下部关节动作较小

这促使后续重点排查：

- 模型迁移是否正确
- 控制量是否真正作用到了底盘和所有关节
- 奖励设计是否偏向仅靠末端少量运动拿分

## 7. 成功判定与奖励函数调整

### 7.1 初始问题

早期训练中，存在以下问题：

- 只靠接近物体就能拿不少分
- 接触奖励的作用不够突出
- 成功判定过严或过松都会影响学习方向
- 底盘是否移动、机械臂是否全身参与，不够明确

### 7.2 奖励设计调整方向

后续围绕以下奖励项反复修改：

- `r_base_pos`
- `r_ee_pos`
- `r_precision`
- `r_close`
- `r_touch`
- `r_orient`
- `r_stability`
- `r_ctrl`

整体思路是：

1. 减弱不必要的奖励项
2. 强化“末端真正靠近目标”
3. 提高“接触目标”或“极近距离成功”的激励
4. 让奖励更服务于“Tracking 成功率”而不是表面 reward

### 7.3 具体改动

先后做过的关键调整包括：

- 降低姿态奖励权重
- 将姿态奖励改为“夹爪朝向尽量与目标速度方向一致”
- 去掉或减弱 `reward_touch` 在某些阶段的干扰
- 增强 `reward_ee_pos`
- 增强 `reward_ee_precision`
- 调整 `reward_close`
- 去除 `r_stability`
- 重新平衡“接近奖励”和“接触奖励”

### 7.4 成功判定改动

后续为了让 tidybot 更容易学到“碰到目标”，成功条件改为满足任一条件即可：

1. 物体真实接触夹爪 pad geom
2. `ee_distance < 0.02`

这意味着：

- 不再只依赖离散接触
- 只要夹爪末端距离物体足够近，也视为成功

这样做的好处是：

- 减少 MuJoCo 接触离散判定带来的训练抖动
- 让策略更容易获得正反馈

## 8. 训练参数调整过程

### 8.1 并行环境数

训练中曾把：

- `num_envs` 调为 `8`

这样可以提高采样速度，但也会增加 CPU 压力。  
在机器资源不足、同时打开浏览器和其他软件时，曾出现训练卡顿甚至像“卡死”的体验。

### 8.2 mini epochs

曾讨论 `mini_epochs = 2` 是否偏少，后续将其调整为：

- `mini_epochs = 4`

这样做的考虑是：

- 同一批采样数据可以被重复利用更多次
- 有助于提升 PPO 的更新强度
- 但也会增加训练时间

### 8.3 minibatch size

后续将：

- `minibatch_size` 调为 `64`

这一步的主要目的是：

- 提升训练稳定性
- 让梯度更新更细一点
- 避免 batch 过大时学习过于粗糙

### 8.4 当前正式训练参数

截至目前，当前训练配置核心参数为：

- `learning_rate = 1e-4`
- `horizon_length = 64`
- `minibatch_size = 64`
- `mini_epochs = 4`
- `max_agent_steps = 6000000`
- `action_track_denorm = [1.5, 0.015, 0.01]`

这些参数主要记录在：

- `configs/train/DcmmPPO.yaml`

## 9. 抛球逻辑与初始条件调整

### 9.1 早期问题

在训练和测试时曾发现：

- 球还没抛出，机器人就已经开始运动
- 这样会导致策略学到“提前空追”
- 训练和测试的时间对齐感不好

### 9.2 修改目标

后来明确要求：

- 小球抛出和机器人开始响应应尽量同步

### 9.3 调整结果

后续把物体静止悬停时间调整为：

- `object_static = [0.0, 0.0]`

这样当前设定下：

- 物体不再先静止悬停一段时间
- 机器人与来球更接近“同时开始”

这个改动提高了任务定义的一致性，但也带来了一个现象：

- 旧策略在新设定下成功率可能下降

这是正常的，因为：

- 训练任务分布变了
- 策略原本学的是“球先不动，再动”的节奏

## 10. 稳定性、地板和初始姿态修改

### 10.1 机器人抖动问题

在使用 tidybot 后，曾多次观察到：

- 机器人整体抖动
- 机械臂初始姿态不自然
- 测试和遥操作时视觉效果不稳定

### 10.2 对策

后续采取了以下措施：

- 增加地板平面
- 调整控制噪声与动作扰动
- 调整机械臂初始姿态
- 使机器人初始朝向更合理
- 让来球方向与机器人初始朝向更匹配

### 10.3 来球方向与站位对齐

后续还专门做了：

- 底盘初始朝向与来球方向对齐
- 机械臂初始姿态调整为更适合迎球
- 目标初始抛掷方向与机器人“面对来球”的场景匹配

这样做以后：

- 训练初期更容易获得有意义的样本
- 底盘和机械臂分工更自然

## 11. 遥操作与仿真辅助调试

### 11.1 遥操作脚本

为了方便直接观察模型是否能被控制，后续编写并修改了：

- `teleop_tidybot.py`

其目的主要有两个：

1. 用键盘直接操控底盘与机械臂
2. 辅助检查模型、关节、视角和控制链是否正常

### 11.2 遇到的问题

在遥操作和 viewer 过程中遇到过：

- `GLXBadDrawable`
- `NV-GLX missing`
- 段错误
- 画面重复刷新
- 模型不断回到初始位置
- 不想测试时出现抛球和重置

### 11.3 结论

后来逐步明确：

- `DcmmVecEnv.py --viewer` 能打开控制，是因为它直接复用了环境本身的 viewer 逻辑
- `teleop_tidybot.py` 则需要额外适配显示与事件循环

同时也把遥操作场景改成了更适合单独调试的模式：

- 不抛球
- 不自动 reset
- 只看机器人自身运动

这一步虽然不直接提升训练指标，但对排查“模型是否真的被控制住了”非常关键。

## 12. 训练结果阶段记录

### 12.1 早期短测

在很短的测试运行中，曾看到：

- `max_agent_steps` 非常小
- `Current Best` 先从 `-10000` 变到 `0`
- `episode_success_per_step = 0`

这类短测的作用主要是：

- 确认环境是否能启动
- 确认模型前向是否正常
- 确认训练循环没有崩

不能据此判断策略优劣。

### 12.2 一次 300 万步训练

曾有一次较早的 tidybot 训练结果：

- `Agent Steps: 3000K`
- `Current Best: 61.07`
- `metrics/episode_rewards_per_step ≈ 61.05`
- `metrics/episode_success_per_step ≈ 0.71078`

这说明当时已经能把 Tracking 成功率训练到约 `71%`。

### 12.3 一次 700 万步训练

另一轮训练结果中出现过：

- `Agent Steps: 7000K`
- `Current Best: 67.73`
- `metrics/episode_rewards_per_step ≈ 66.98`
- `metrics/episode_success_per_step ≈ 0.17637`

这说明：

- reward 很高不等于 success 一定高
- 当奖励设计与真实目标不完全一致时，策略可能学会“拿分”但不一定真正追得更好

这一现象促使后续继续修改奖励设计。

### 12.4 一次 1000 万步训练

又有一轮训练结果为：

- `Agent Steps: 10000K`
- `Current Best: 38.29`
- `metrics/episode_rewards_per_step ≈ 35.04`
- `metrics/episode_success_per_step ≈ 0.47609`

这进一步说明：

- 不是步数越多越一定好
- 当任务分布、成功判定或奖励结构改变后，需要重新评估训练策略

### 12.5 一次 600 万步训练

后面有一轮较为成功的训练结果为：

- `Agent Steps: 6000K`
- `Current Best: 52.09`
- `metrics/episode_rewards_per_step ≈ 49.99`
- `metrics/episode_success_per_step ≈ 0.80638`

这一结果说明：

- 经过奖励和平衡调整后，Tracking 成功率重新回到较好水平
- 这版训练结果比一些高 reward 但低 success 的版本更可靠

## 13. 从“真值观测”转向“视觉观测”

### 13.1 转向视觉方案的原因

后续项目目标不再停留在仿真里的“直接给 PPO 物体真值坐标和速度”，而是希望更贴近真实机器人部署。  
因此引入了新的要求：

- 使用底座上的相机观测目标
- 相机识别物体的三维位置和速度
- 再把这些信息传给 PPO

这样做的意义在于：

- 更接近真实机器人感知链路
- 为后续真机部署做准备

### 13.2 传感器方案讨论

曾讨论过的硬件包括：

- DJI Action 5
- RealSense D455i

最终在仿真中先采用“RGB-D 相机 + 视觉估计”的路线。

### 13.3 视觉链路设计

后续在项目中加入了完整的视觉状态估计链路：

```text
base 相机 RGB-D
  ->
红色目标检测
  ->
深度反投影
  ->
得到 3D 位置
  ->
3D 跟踪器估计速度
  ->
传给 PPO
```

相关新增或修改的模块包括：

- `gym_dcmm/vision/detector.py`
- `gym_dcmm/vision/depth_projector.py`
- `gym_dcmm/vision/tracker3d.py`
- `gym_dcmm/vision/visual_state_estimator.py`

## 14. 视觉链路调试过程

### 14.1 初始视觉误差较大

在视觉链路刚接通时，曾打印出类似这样的误差：

- 位置误差约 `1.4m ~ 2.6m`
- 速度误差约 `1.4m/s ~ 3.5m/s`
- 估计的高度方向甚至会出现明显错误

这说明当时虽然“看起来已经在做视觉估计”，但实际世界坐标投影并不对。

### 14.2 关键 bug：动态相机外参没有实时刷新

后来定位到一个非常关键的问题：

- 对于 `targetbody` 相机，不能一直用静态 XML 外参
- 必须实时使用 MuJoCo 里的 `data.cam_xmat` 和 `data.cam_xpos`

后续在 `MujocoDcmm.pixel_2_world()` 里修复了这一点。  
这是视觉链路中最重要的一次修复之一。

修复后，位置误差显著下降。

### 14.3 速度跟踪器改进

后续又把 `tracker3d.py` 从“常速度模型”改成了更适合抛物线目标的：

- 常加速度 / 带重力模型

修改后带来的直接改善是：

- 速度误差明显下降
- 速度方向更加稳定

## 15. 相机位置与视场角实验

### 15.1 为什么要改相机位置

在视觉方案里，相机位置会直接影响：

- 是否容易被机械臂遮挡
- 能否完整看到来球轨迹
- 深度反投影的稳定性

因此后续围绕 base 相机做了多轮实验。

### 15.2 视场角实验

曾对以下视场角做过对比：

- `fovy = 55`
- `fovy = 60`
- `fovy = 65`

最终结论是：

- `65` 最稳
- `55` 偏窄，容易丢检
- `60` 居中，但整体仍略弱于 `65`

因此最终保留：

- `fovy = 65`

### 15.3 相机安装位置实验

后续又在“中线位置”和“侧边位置”之间做了对比。  
当前最终保留的相机位置为：

- `(-0.089, -0.214, 1.600)`

这是一个安装在底盘侧后方立杆上的 base 相机方案。

### 15.4 位置实验结果

在 30 回合对比中，得到如下趋势：

- 侧边方案平均位置误差约 `0.020 m`

## 17. 视觉训练模式切换

### 17.1 中间阶段

最开始的视觉训练并不是完全纯视觉，而是：

- PPO 吃视觉位置
- 暂时不直接吃视觉速度
- 姿态奖励仍可能借用真值速度做 shaping

这样做的目的是先保证训练稳定。

### 17.2 当前正式模式

后来决定切到当前正式训练模式：

- `use_visual_object_state = True`
- `use_visual_object_velocity = True`
- `fallback_to_ground_truth = False`

即：

- PPO 的物体位置来自视觉
- PPO 的物体速度来自视觉
- 不再把物体真值状态直接喂给 PPO

需要说明的是：

- 当前 reward 和 success 判定仍然基于仿真真值
- 但策略输入已经是纯视觉物体状态

这在强化学习训练中是合理的，因为：

- 观测负责模拟真实感知
- 奖励负责提供训练监督
